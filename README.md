# Аналіз Spotify Tracks у MongoDB

Проєкт завантажує CSV-набір треків у MongoDB, перетворює сирі записи на
структуровані документи та виконує запити, агрегації й аналіз індексів.

## Структура проєкту

- `scripts/01_load_data.py` — завантаження CSV у колекцію `tracks_raw`;
- `scripts/02_transform.js` — перетворення даних і створення колекції `tracks`;
- `queries/part2_queries.js` — вибірки та агрегації частини 2;
- `queries/part3_aggregations.js` — агрегації частини 3;
- `queries/part4_indexes.js` — створення індексів та перевірка через `explain()`.

## Налаштування оточення

### Передумови

Потрібні:

- Python 3.10 або новіший;
- `mongosh`;
- MongoDB Atlas або локальний MongoDB-сервер;
- файл `dataset.csv` із набору Spotify Tracks, розміщений у
  `~/Downloads/archive/dataset.csv`.

Шлях до CSV заданий константою `CSV_PATH` у
[`scripts/01_load_data.py`](scripts/01_load_data.py). Якщо файл знаходиться в
іншому каталозі, цю константу потрібно змінити.

### Встановлення залежностей

```bash
python3 -m venv .venv
source .venv/bin/activate
python -m pip install -r requirements.txt
```

### Налаштування `.env`

У корені проєкту потрібно створити файл `.env`:

```dotenv
MONGO_URI="mongodb+srv://<username>:<password>@<cluster>/?retryWrites=true&w=majority"
```

Файл `.env` доданий до `.gitignore`, тому облікові дані не потрапляють у Git.
Для Atlas також потрібно дозволити підключення своєї IP-адреси в Network
Access і надати користувачеві права на базу `spotify`.

### Порядок запуску

Команди виконуються з кореня репозиторію:

```bash
# Експортувати змінні з .env для mongosh
set -a
source .env
set +a

# 1. Завантажити CSV у spotify.tracks_raw
python scripts/01_load_data.py

# 2. Побудувати підсумкову колекцію spotify.tracks
mongosh "$MONGO_URI" --file scripts/02_transform.js

# 3. Виконати завдання частин 2–4
mongosh "$MONGO_URI" --file queries/part2_queries.js
mongosh "$MONGO_URI" --file queries/part3_aggregations.js
mongosh "$MONGO_URI" --file queries/part4_indexes.js
```

Якщо змінна `MONGO_URI` не експортована в поточну оболонку, URI можна передати
безпосередньо:

```bash
mongosh "mongodb+srv://<username>:<password>@<cluster>/" \
  --file scripts/02_transform.js
```

Скрипт завантаження видаляє `tracks_raw` перед імпортом, а трансформація
перезаписує `tracks` через `$out`, тому їх можна запускати повторно.

## Схема колекції `tracks`

Підсумкова структура створюється стадією `$project` у
[`scripts/02_transform.js`](scripts/02_transform.js):

```json
{
  "_id": "ObjectId",
  "track_id": "string",
  "track_name": "string",
  "album_name": "string",
  "artists": ["string"],
  "explicit": false,
  "popularity": 73,
  "duration_ms": 214000,
  "track_genre": "pop",
  "popularity_tier": "high",
  "audio_features": {
    "danceability": 0.76,
    "energy": 0.81,
    "loudness": -5.4,
    "mode": 1,
    "speechiness": 0.05,
    "acousticness": 0.12,
    "instrumentalness": 0.0,
    "liveness": 0.09,
    "valence": 0.68,
    "tempo": 124.1,
    "key": 5,
    "time_signature": 4
  }
}
```

`popularity_tier` обчислюється так:

- `high` — `popularity >= 70`;
- `medium` — `40 <= popularity < 70`;
- `Low` — `popularity < 40`.

`artists` утворюється поділом початкового рядка за `;`, а
`audio_features` об’єднує всі числові характеристики звучання.

## Завдання 1

### 1. Чому аудіо-характеристики винесені в окремий об’єкт audio_features, а не зберігаються плоско? Коли таке вкладення вигідне, а коли створює проблеми?

Аудіо-характеристики логічно обєднуються в одну групу, так як вони описують конкретні (аудіо) характеристики треку. Логічний об'єкт `audio_features` створює namespace для всієї групи. Це зручно для розуміння документів, адже ми бачитимемо не обєкт з десятками полів, а деревовидну структуру яку легко зрозуміти. Також, це полегшує проекцію у запитах, адже можна одразу всі поля віднесені до одного оюєкту відключити (audio_features: 0). Якщо у майбутньому добавляться нові поля у цей обєкт, це не вплине на проекції верхнього рівня документу. З недоліків я бачу те, що запити, проекції стають довшими через нотацію з крапкою: "audio_features.mode" vs "mode", проте це косметичний недолік.

### 2. Чому виконавці зберігаються як масив, а не як рядок? Які запити стають простішими?

У випадку зберігання виконавців як массив ми можемо легко оперувати окремими виконавцями: агрегувати, фільтрувати, сортувати по окремим виконавцям. Усі запити які стосуватимуться окремого виконавця стають простішими. У випадку зберігання одним рядком потрібно буде використовувати regexp, що значно ускладнить і уповільнить запити (адже не можливо буде налаштувати пошукові індекси по окремим артистам).

### 3. Що таке $out і чим він відрізняється від $merge? Коли використовувати кожен?

Це оператори aggregation pipeline, обидва використовуються для збереження результатів pipeline у колекці. Основна різниця полягає у тому, що $out повністю видаляє стару колекцію і створює нову. $merge - додає нові документи та оновлює існуючі, документи які не були у pipeline, але є у колекції - не чіпає. Тому $out використовується коли потрібно повністю замінити колекцію не зважаючи, що було у ній раніше (от як у нашому прикладі), побудова якигось "вітрин даних" чи кешів (для прикладу: top N покупок). $merge використовується для інкрементального оновлення існуючих колекцій новими даними.

## Завдання 2

### 1. Для чого використовується інструкція $unwind?

Це оператор aggregation pipeline, який використовується для розгортання массивів, тобто, у наступний оператор документ буде продубльований стільки ж разів, скільки елементів у вказаному массиві, але замість массива буде відповідне значення з массива. Це необхідно для  роботи з елементами масиву (групування за елментами масиву чи сортування).

### 2. Чим $stdDevPop відрізняється від $stdDevSamp?

Обидва оператори використовуються для обчислення стандартного відхилення, проте використовують різні формули для її підрахунку. $stdDevPop - використовується коли наші дані це генеральна сукупність (є всі дані які нас цікавлять, не окрема випадкова вибірка), $stdDevSamp у випадку, якщо у нас лише невелика випадкова вибірка (для прикладу, топ 50/100 треків по жанрам) на основі якої ми хочемо зробити висновок по глобальні тренди. У другому випадку, у формулі в знаменнику N-1 (к-сть елементів).

## Завдання 3

### 1. У запиті 1 ми фільтруємо виконавців, у яких менше 5 треків. Як зміниться результат, якщо знизити поріг до 1? А що станеться, якщо вибирати виконавців із більш ніж 50 треками? Поясніть результат.

Якщо знизити поріг до 1 треку повністю зміниться список топ 10, але цей результат буде не репрезентативний, так як у виконавця може бути всього лише 1 трек, і такий артист знаходитиметься у списку значно вище за артистів, які мають трек з такою ж популярістю, але ще мають інші треки, які дещо менш популярні. Також, змінивши поріш до 50 треків, у список попадуть артисти, які ведуть свою діяльність уже багато років, артисти, які ще не встигли випустити 5-7 альбомів не матимуть можливості попасти у рейтинг. Тому, обмеження у 5 означає, що  у артиста уже є 1 альбом, і не один трек був популярним.

### 2. У запиті 3 ми фільтруємо жанри з менше ніж 100 треками. Чи зміниться результат, якщо знизити поріг до 50? Поясніть результат.

Результат не змінився, тому що у вибірці найменший жанр з 999 треками (k-pop), отож, даний поріг у 100 треків не впливає на результат.

### 3. Чи є цей запит покривним (covered query)? Надайте розгорнуту та обґрунтовану відповідь у файлі README.

```javascript
db.tracks.find({
    track_genre: "pop",
    popularity: { $gte: 70 }
})
```

Даний запит не є покривним, тому що, для того щоб він був покривним, повинні виконатись 2 умови:

1. Усі поля, що є у фільтрації є частиною індекса (ця умова виконана, індекс з першого завдання це покриває)
2. Усі поля у проекції є частиною індексу (ця умова не виконана, так як у запитів повертаються усі поля), і mongo engine не буде потрібно читати інформацію по полям з диска, а одразу їх отримає з індекса, який часто є підвантаженим у RAM.

Для того, щоб зробити запит покривним потрібно добавити проекцію:

```javascript
{
    _id: 0, // обовязково, бо цього поля немає у індексі з завдання 1
    track_genre: 1,
    popularity: 1
}
```
